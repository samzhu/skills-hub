## ⛔ Cancellation Note 2026-05-13

1. **Backend 無 `skills.tags` column** — `V1__initial_schema.sql:44-56` skills 表只有 `id/name/description/author/category/latest_version/risk_level/status/download_count/created_at/updated_at`。整個 `db/migration/` 目錄 grep "tags" → 0 hit；V2~V22 都沒加。
2. **Backend Skill domain 也無 tags 欄位** — `backend/src/main/java/.../skill/` 整 package grep "tags" → 0 hit。`SearchIntentService.java:17` 那個 "tags" 是 LLM 即時生成 4 個英文 concept tags 給搜尋頁顯 chip，跟 skill 本身無關。
3. **Frontend Skill type 無 tags 欄位** — `frontend/src/types/skill.ts:30-78` Skill interface 沒有 `tags` field；`SkillCard.tsx` 也沒任何 tag 字串。
4. **Frontend HomePage 無 tag chip UI** — 現有 filter 只有 `RiskFilterSidebar` + `CategorySidebar` + 4 個 sort chips。沒 tag chip → spec §1「點了 chip 沒效」的症狀不存在。
5. **SkillQueryController 簽名已漂移** — spec §2.2 寫 `search(category, tag, search, pageable)`；實際（S159a/b/d 之後）是 `search(keyword, category, author, pageable)`。

S159 META §2.2 的 origin observation「frontend 顯示 tag」當初也是誤觀察。

**未來若真要加 per-skill tags**，是新 feature spec：要決定 tags 怎麼存（DB column / 獨立 tag table / SKILL.md frontmatter parse + cache）、誰能編輯、UI 編輯流程、回填策略。先確認 PRD 真的需要再開新 spec，不要復活本 spec。

## 1. Goal

**一句話：** 讓 `GET /api/v1/skills?tag=devops` 真的回 tag 含 "devops" 的 skill list（目前 backend ignore，silently 回全部）。

**為什麼重要：**
- 平台已有 `skills.tags` JSONB 欄位（per V1 schema）
- 前端 SkillCard 顯 tags、frontend filter UI 已備（但點了 chip 沒效）
- backend `SkillQueryController.search()` 不接 `tag` param → silently dropped → user 體感「filter 壞了」

**非目標：**
- 不做 multi-tag AND / OR（本 spec 單 tag filter 足；多 tag combine 留 follow-up）
- 不做 tag autocomplete（前端已有 chip 列表夠用）

## 2. Approach

### 2.1 現況

`skills.tags JSONB` 存 `["devops", "k8s", "docker"]` array string。`SkillQueryController.search()` 接 `category / search` 兩個 query param，**無 `tag` param**。`SkillRepository` 只有 `findAll`/`findByCategory` 等 derived，**無 findByTag**。

PostgreSQL JSONB 查 array contains：`tags @> '["devops"]'::jsonb` 或 `tags ?? 'devops'`（key existence）。

### 2.2 設計

**Backend：**

```java
// SkillQueryController
@GetMapping("/skills")
public Page<Skill> search(
    @RequestParam(required = false) String category,
    @RequestParam(required = false) String tag,         // NEW
    @RequestParam(required = false) String search,
    Pageable pageable
) {
    return skillQueryService.search(category, tag, search, pageable);
}

// SkillQueryService — 既有 raw SQL 加 WHERE clause
String sql = """
    SELECT * FROM skills WHERE 1=1
    AND (:category IS NULL OR category = :category)
    AND (:tag IS NULL OR tags @> :tagJsonbArray::jsonb)        -- NEW
    AND (:search IS NULL OR (name ILIKE :searchPattern OR description ILIKE :searchPattern))
    ORDER BY ...
""";
// :tagJsonbArray = '["' + tag + '"]'  （single-element array literal）
```

對齊 PostgreSQL JSONB containment operator `@>`，可用 GIN index `idx_skills_tags ON skills USING GIN(tags jsonb_path_ops)` 加速（**檢查既有是否有 — 若無加 V20 migration**）。

**Frontend：**

`HomePage` filter chip 點擊 → URL query `?tag=devops` → `useSkillList({ tag })` → API call。**前端可能已有此邏輯** — 需 verify。

### 2.3 Decision points

- **AND/OR 多 tag**：本 spec 單 tag。若未來要 `?tag=devops&tag=k8s` AND filter，加 `@RequestParam List<String> tag` + `tags @> :tagsJsonbArray`（traversal contains all）— 留 follow-up
- **GIN index**：若既有無，本 spec 加 V20；否則 skip migration

## 3. Acceptance Criteria

```
AC-1: ?tag=<existing> 回對應 skill
  Given 平台有 skill A (tags=["devops","k8s"]) + skill B (tags=["docs"])
  When GET /api/v1/skills?tag=devops
  Then 回 [skill A]（不含 B）

AC-2: ?tag=<non-existent> 回空
  Given 同上
  When GET /api/v1/skills?tag=nonexistent
  Then 回 []（200 + 空 page）

AC-3: ?tag 與 ?category 可組合
  Given skill A (category=DevOps, tags=[devops]); skill C (category=DevOps, tags=[docker])
  When GET /api/v1/skills?category=devops&tag=devops
  Then 回 [skill A]（不含 C）

AC-4: 無 ?tag 時不影響原行為
  Given 任何 skill set
  When GET /api/v1/skills（無 tag param）
  Then 回全部（不 filter）

AC-5: tag case-sensitive 對齊現況（tags JSONB 存什麼就比什麼）
  Given skill A tags=["DevOps"]（大寫）
  When GET /api/v1/skills?tag=devops（小寫）
  Then 回 []（不 match）— 對齊現況不做 normalize（若要 normalize 開新 spec）

AC-6: GIN index 存在或新加（若需）
  Given V20 migration 加 idx_skills_tags（若 V1 無）
  When 跑 EXPLAIN ANALYZE SELECT WHERE tags @> ...
  Then 用 GIN index seek（不全表掃）

AC-7: 前端 filter chip 點擊後 URL 同步 + 結果正確
  Given HomePage 顯示 tag chip "devops"
  When user 點擊
  Then URL 更新 ?tag=devops + skill list re-fetch + 顯示 filtered result
```

**驗證指令：** `cd backend && ./gradlew test` + `cd frontend && npm test`

# S024-T01: Infrastructure + Skill aggregate skeleton + cross-aggregate POC RED test

## Spec
S024 — Skill State-Based Aggregate Migration

## Goal
建立 S024 的最低基礎建設讓 Skill aggregate 能透過 `repo.save()` 觸發 `@DomainEvents` 進入 Modulith outbox，並用 first failing test 同時驗 §2.9 POC hypothesis（跨 aggregate 同 TX publish 兩 aggregate 的事件至 outbox）。

POC 折進本 task 的 RED test（per Phase 1 決策；避免 duplicate setup）。若 RED→GREEN 失敗 → escalate `/planning-spec S024` 改設計，**不**繼續推 T2-T6。

## Anti-Patterns / Pitfalls

- **不要**先刪 SkillReadModel — 留至 T5；T1-T4 期間 read side 仍走舊 read model（保留 SkillProjection sync `@EventListener` 等舊路徑），新 Skill aggregate 與舊 SkillReadModel 並存（兩個都對應 skills 表，但 ReadModel 的 `Persistable.isNew()=true` 永遠 INSERT，與 aggregate 走 isNew=auto 衝突）
  - 解：`SkillReadModel` 用 `@Table("skills")` 同表；T1 期間先讓 Skill 也 `@Table("skills")` — 兩個 @Entity 對同表，Spring Data JDBC 不衝突（converter 各自 routing）
  - **但**：T1 RED test 內呼叫 `skillRepo.save(Skill.create(...))` 寫一筆 row；既有 SkillProjection（sync `@EventListener`）也會寫一筆 — 會撞 PK 衝突
  - **解法**：T1 期間 SkillProjection.on(SkillCreatedEvent) 先標 `@ConditionalOnProperty(name = "skillshub.s024.legacy-projection-enabled", havingValue = "true", matchIfMissing = true)` → 在 T1 test profile 設 false 關掉舊 listener 避免衝突；T5 才整檔刪除
  - 或：T1 RED test 不依賴舊 SkillProjection；用獨立 test profile
- **不要**讓 `SkillVersionPublishedEvent.of()` 自動產生新 sourceEventId 在 retry 時改變（既有 record 有 `sourceEventId` field；保留語義）
- 不要把 `Skill.fromHistory` 改 public — `@Deprecated` + `private static` 才符合 spec §4.1 emergency rollback 設計
- POC 預期 listener 數量寫死為 3 是 spec §2.9 估計值；實際以 T1 跑出的真值為準（記錄至 task result + spec §6 POC Findings）

### POC Findings (spec §6 POC Findings — 寫入 spec 後刪本段)

**Hypothesis**：跨 aggregate 同 TX `@DomainEvents` publish 行為（per spec §2.9）— **VALIDATED**。

**實際 event_publication outbox row count**: **3 rows**（per RED test log）

**實際 listener_id 列表**:
1. `io.github.samzhu.skillshub.search.SearchProjection.onSkillCreated(...SkillCreatedEvent)` → 訂閱 `SkillCreatedEvent`
2. `io.github.samzhu.skillshub.search.SearchProjection.onVersionPublished(...SkillVersionPublishedEvent)` → 訂閱 `SkillVersionPublishedEvent`
3. `io.github.samzhu.skillshub.security.scan.ScanOrchestrator.on(...SkillVersionPublishedEvent)` → 訂閱 `SkillVersionPublishedEvent`

**實際 distinct event_type 列表**:
- `SkillCreatedEvent` — 1 row（SearchProjection 1 listener）
- `SkillVersionPublishedEvent` — 2 rows（SearchProjection + ScanOrchestrator）

**SkillVersionPublishedFromAggregate**: 0 rows（T1 無 subscriber；T5 加 AuditEventListener 訂閱後才出現於 outbox — 預期行為）

**Spec §2.9 預測 vs 實際**:
- spec 預期 3 listener entries（SearchProjection / ScanOrchestrator / AuditEventListener × `SkillVersionPublishedEvent`）
- 實際 3 listener entries — 數量正確；但分布為「SearchProjection × 2 (跨兩 event class) + ScanOrchestrator × 1」而非 spec 預設「3 listeners 訂閱 SkillVersionPublishedEvent」
- AuditEventListener 在 T5 才實作 — 實作後預期 outbox row 增至 9 rows（每個 event class × 對應 listener）

**Cross-aggregate same TX publish 行為 — 確認**:
- 同 `@Transactional` 內 `skillRepo.save(skill)` + `skillVersionRepo.save(version)` 各自的 `@DomainEvents` 觸發 publish；event_publication INSERT 與業務 SQL 同 TX
- TX rollback 測試 — 業務 throw 後 skills / skill_versions / event_publication 三表皆無對應 aggregate_id 的 row
- AFTER_COMMIT async listener 確實 fire（log 證實 SearchProjection / ScanOrchestrator 三 method 各自被調用），但部分因 test 環境缺 storage / embedding model 而拋例外 → outbox row 標 status=FAILED（per S023 設計）— 不影響 POC 結論

**結論**：T2-T6 task plan 設計可繼續推進；spec §2.9 hypothesis VALIDATED；無需 escalate `/planning-spec` 改設計。

### Notes / 觀察 / Drift

1. **Persistable.isNew() 必要性**：spec §4.1 未明確要求 implements Persistable；實作中發現 @Id 非 null 預設走 UPDATE path（per deepwiki §4.isNew + §1.@Version 陷阱）— 必須自訂 isNew 才能讓 first save 走 INSERT。
   - Skill：用 `@Version Long version` 是否 null 判斷
   - SkillVersion：用 @Transient `boolean isNew` flag（factory 設 true、@PersistenceCreator path 載入維持 false 走 UPDATE 給 T3 attachRiskAssessment）
   - 建議寫入 spec §4.1 / §4.3 或合併進 T2/T3 task notes 與 development-standards.md「Spring Data JDBC Rich Aggregate 規範」（T6）

2. **Legacy SkillProjection sync handler 不用 @ConditionalOnProperty 切換**：原 task plan 建議 `@ConditionalOnProperty` legacy switch；實作改用 `existsById` runtime gate — 更乾淨（無 test profile 設定、無新 cache key buster）。Production 路徑（ES path 透過 eventStore.save 寫 domain_events，不寫 skills row）下 existsById=false 持續走原路徑；新 path 下 existsById=true 跳過。T5 整檔刪 SkillProjection 後本 gate 隨之消失。

3. **S016 owner ACL seed 邏輯下放至 Skill.create 工廠**：原 SkillProjection.on(SkillCreatedEvent) 在 listener 中組 owner ACL；現 Skill.create 直接 seed `aclEntries` field（與 V2 backfill / SkillProjection 同邏輯），保證新 path 與舊 path ACL 行為一致。

4. **ScanOrchestrator log 顯示 InMemoryStorageService not-found exception**：T1 POC test 用假 storagePath `gs://bucket/cross-tx/1.0.0.zip`；test 環境 InMemoryStorageService 無此 file → ScanOrchestrator async listener 拋 RuntimeException → 由 listener 內 try-catch 吞掉（per S010 設計：scan pipeline 失敗不阻塞業務）→ outbox status=FAILED。**不影響 POC 結論**（POC 驗 outbox INSERT 行為，不驗 listener 完成度）。

5. **SkillVersionPublishedFromAggregate 預期 0 listener 訂閱**：T5 加 AuditEventListener.on(SkillVersionPublishedFromAggregate) 訂閱後 outbox row 增加。T1 RED test 不對此事件 row 數量做 assertion（避免錯依賴未實作的 T5 行為）。

6. **Compile warnings**：`SkillCommandService.java uses or overrides a deprecated API`（既有 `Skill(String, List<DomainEvent>)` constructor 與 `Skill.create(String, String, String, String)` 都標 @Deprecated）— T4 移除呼叫端後此 warning 消失。

7. **Spec §4.1 中 Skill 列出 `riskLevel` 欄位** 但 ADR-002 §2.1 表格中改寫前 SkillReadModel 含 `risk_level` 欄位 — 我在 Skill.java 保留 `@Column("risk_level") String riskLevel`（schema column 存在；S010 ScanOrchestrator 仍寫此值）。T5 ScanOrchestrator 改造會處理 cross-aggregate projection 細節。

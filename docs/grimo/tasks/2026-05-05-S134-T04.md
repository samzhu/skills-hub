# S134-T04: Manual E2E smoke — real IdP login flow + 真實 claim shape 紀錄

## Spec
S134 — Real OAuth IdP Local Dev Integration Trial

## Role
**This task IS the POC** — first real-IdP login flow validates spec §2.3 兩個 Hypothesis：
1. IdP-side PKCE 接受 confidential client（Spring Security 7 default `requireProofKey=true`）
2. 真實 IdP claim shape（advertised metadata 缺 `claims_supported`，登入後才知道）

## BDD

```gherkin
Scenario: AC-2 — discovery 302 含 PKCE + state
  Given backend 跑 SPRING_PROFILES_ACTIVE=local,dev,real-oauth ./gradlew bootRun
  And application-real-oauth.yaml 已填好 client_id + client_secret
  When curl -s -i http://localhost:8080/oauth2/authorization/skillshub
  Then 回 302
  And Location header 為 https://auth-dev.omnihubs.cloud/oauth2/authorize?<query>
  And query 含 client_id=596527ca-e045-4cbe-b78b-692c9a303e14
  And query 含 redirect_uri=http%3A%2F%2Flocalhost%3A8080%2Flogin%2Foauth2%2Fcode%2Fskillshub
  And query 含 scope=openid
  And query 含 code_challenge=<base64url> + code_challenge_method=S256（PKCE 啟用）
  And query 含 state=<random>（CSRF 防護）

Scenario: AC-3 — 完整 login flow（瀏覽器手動跑）
  Given AC-2 通過；developer 開無痕分頁打 http://localhost:8080/oauth2/authorization/skillshub
  When 完成 IdP 登入頁輸入 credentials → 點 confirm
  Then IdP 302 回 http://localhost:8080/login/oauth2/code/skillshub?code=...&state=...
  And backend 換 token 成功（log 不出現 InvalidClientException / InvalidGrantException / IllegalStateException）
  And 最終 browser landing 在 http://localhost:8080/（首頁＝搜尋頁）
  And browser DevTools Application tab 看到 JSESSIONID cookie

Scenario: AC-4 (real) — `/api/v1/dev/auth-debug` 回真實 IdP token claim shape
  Given AC-3 完成（session cookie 仍在 browser）
  When 同 browser GET http://localhost:8080/api/v1/dev/auth-debug 或 /auth-debug 頁
  Then 回 200 JSON
  And id_token_claims.iss == "https://auth-dev.omnihubs.cloud"（驗 trailing-slash 對齊）
  And id_token_claims.sub 非 null 非空
  And access_token_claims 為 decoded JWT payload Map
  And 把完整 JSON 紀錄到 spec §7（mask client_id 後 8 字元 + 整段 access_token_value）

Scenario: AC-5 — bearer token 路徑同 issuer 也運作（dual-path）
  Given AC-4 完成；developer 從 access_token_value 拿到完整 access_token
  When curl -H "Authorization: Bearer <access_token>" http://localhost:8080/api/v1/dev/auth-debug（無 session cookie）
  Then 回 200 JSON
  And principal_name == JWT 的 sub
  And 此驗證確認 oauth2Login + oauth2ResourceServer 同 chain 對 same issuer JWT 兩路皆過

Scenario: AC-6 (final regression) — 預設 dev profile 不受影響
  Given application-real-oauth.yaml 已存在
  When SPRING_PROFILES_ACTIVE 不含 real-oauth → ./gradlew bootRun
  Then `/api/v1/skills` anonymous 200（既有 LAB 行為）
  And `/api/v1/dev/auth-debug` 404（@Profile guard 沒 register bean）
  And mock-oauth2-server compose container 正常啟動
```

## Execution Steps（user 操作）

```bash
# Step 1：filled-in real-oauth profile 設定檔（user 端，不 commit）
cp backend/config/application-real-oauth.yaml.example \
   backend/config/application-real-oauth.yaml
# 編輯 backend/config/application-real-oauth.yaml 填入完整 client_secret

# Step 2：啟動 backend（real-oauth profile）
cd backend
SPRING_PROFILES_ACTIVE=local,dev,real-oauth ./gradlew bootRun

# Step 3：AC-2 — 驗 discovery 302
curl -s -i http://localhost:8080/oauth2/authorization/skillshub | head -20
# 預期：HTTP/1.1 302 + Location: https://auth-dev.omnihubs.cloud/oauth2/authorize?...
# 觀察 query 含 code_challenge + state + scope=openid

# Step 4：AC-3 — 完整 login flow（瀏覽器無痕分頁）
# 1. 開無痕 → http://localhost:8080/oauth2/authorization/skillshub
# 2. IdP 登入頁輸入 credentials
# 3. 預期 302 回 callback → land 在 http://localhost:8080/

# Step 5：AC-4 — claim shape dump
# 同瀏覽器 → http://localhost:8080/auth-debug 或 GET /api/v1/dev/auth-debug
# 把 JSON 整段複製貼到 spec §7（mask access_token_value 中段，client_id 部分 mask 後 8 字元）

# Step 6：AC-5 — bearer dual-path
ACCESS_TOKEN=<從 step 5 拿到的 access_token_value>
curl -s -H "Authorization: Bearer $ACCESS_TOKEN" \
  http://localhost:8080/api/v1/dev/auth-debug | jq .
# 預期：principal_name == sub claim；無 session 也通

# Step 7：AC-6 final regression — 切回預設 profile
# Ctrl-C step 2；改跑 ./gradlew bootRun（無 profile 變數）
curl -s -i http://localhost:8080/api/v1/skills | head -5  # 預期 200
curl -s -i http://localhost:8080/api/v1/dev/auth-debug    # 預期 404 (bean 不註冊)
```

## Hypothesis Validation Checklist

- [ ] **H1：IdP-side PKCE accepts confidential client**
  - PASS criterion：AC-3 的 token exchange 不出 `invalid_request` / `invalid_grant` 錯誤
  - FAIL fallback（per spec §2.4 #1）：在 `SecurityConfig` 對該 ClientRegistration 加 programmatic override：
    ```java
    .clientSettings(ClientSettings.builder().requireProofKey(false).build())
    ```
    然後重跑 AC-3。記錄為 spec §7 deviation。

- [ ] **H2：真實 IdP claim shape captured**
  - 紀錄 spec §7：完整 JSON dump（masked secrets）
  - 對比 mock-oauth2-server claims（`backend/config/oauth-mock-config.json`）→ 列出差異（哪些 claim 真 IdP 有而 mock 沒、反之亦然）
  - 列出 follow-up spec scope（claim → role mapping 的對齊方式）

## Records to capture（補進 spec §7）

1. **AC-2 curl output**：完整 302 + Location（mask client_id 後 8 字元）
2. **AC-3 backend log snippet**：login 成功的 4-5 行 log（filter chain 處理痕跡）
3. **AC-4 JSON dump**：full claim payload（mask access_token_value mid-段如有需要 + client_id 後 8 字元）
4. **AC-5 curl output**：bearer 路徑同 endpoint
5. **AC-6 regression check**：`/api/v1/skills` 200 + `/api/v1/dev/auth-debug` 404
6. **Mock vs Real IdP claim 對比表**：哪些 claim 名稱對齊 / 不對齊
7. **Hypothesis verdict**：H1 + H2 各 PASS / FAIL + 解析

## BDD
```
AC-5

Given 既有 docs/grimo/architecture.md 只在 dependency table line 535 提
      `org.graalvm.buildtools.native 0.11.5`，無策略性說明 GraalVM AOT 怎麼用
      / 為何用 / 既知限制

When 加新段「GraalVM AOT Strategy」（建議插入位置：「PostgreSQL Configuration」段
     之前，或「Frontend Architecture」之後 — 屬 backend infra）

Then 段內含 5 個要點：
  (a) Production deploy mode：目前 Paketo JVM buildpack（非 native-image），但
      `./gradlew nativeCompile` 可手動跑出 223 MB native binary（POC v4 已驗）
  (b) AOT processing 啟用機制：
      - `shared/aot/AotStubConfig.java` (@Profile("aot")) — 提供 dataSource bean
        繞過 Spring Boot 4 AOT @ConfigurationProperties eager-bean binding 失效
      - `shared/persistence/JdbcConfiguration.jdbcDialect()` override
        — issue spring-boot#47781 workaround
      - `application-aot.yaml` — flyway.enabled=false + GcpContextAutoConfiguration exclude
      - `build.gradle.kts ProcessAot` — args("--spring.profiles.active=aot,local")
  (c) Reflection hint fast-fail 機制：
      - `--exact-reachability-metadata=io.github.samzhu.skillshub` flag (gated by
        `-PexactReachability=true` Gradle property; build.gradle.kts ~line 196)
      - 用法：`./gradlew nativeCompile -PexactReachability=true`
      - 預設 reporting mode = Throw — 任何 missing reflection registration
        在 build 階段直接 fail
  (d) 既知 blocker：cyclonedx-bom 3.2.4 與 Gradle 9.4.1 nativeCompile task graph
      衝突（processResources ↔ cyclonedxBom 互相依賴 race）— 必須先 fix 才能
      ship native production deploy。追蹤：S148f
  (e) 未來啟用 native production deploy 觸發條件：
      - S148f cyclonedx fix ship
      - Cloud Build pipeline 切 BP_NATIVE_IMAGE=true
      - 評估 native binary cold start 收益 vs JVM AOT cache 收益（目前 JVM mode
        已 cold start ~0.5s — 切 native 邊際效益待評估）

And reviewer 讀後能回答：
  - "目前 production 跑 native 還是 JVM？"
  - "誰的 reflection metadata 是手動加 vs auto-register？"
  - "想跑 native compile 怎麼跑？fast-fail 怎麼開？"
  - "為何不直接切 native production deploy？"
```

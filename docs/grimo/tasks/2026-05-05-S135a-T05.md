# S135a-T05: QualityScoreService + Listener + qualityExecutor + Scenario Test

## Spec
S135a — Backend Quality Score（async orchestration 層；3-axis 評分 + 寫 3 rows + sourceEventId idempotency + 獨立 executor pool + 失敗 re-throw → outbox retry）

## BDD

```
Given skill `demo-skill` v1.0.0 publish 觸發 SkillVersionPublishedEvent
When QualityScoreListener.on(event) async 處理（after 5-15s）
Then skill_scores 表有 3 rows
And 每 row 含 axis ∈ {VALIDATION, IMPLEMENTATION, ACTIVATION}
And axis=VALIDATION row 的 dimensions JSONB 含 line_count / body_present 等 rule 結果
And axis=IMPLEMENTATION row 的 dimensions JSONB 含 4 dims (Conciseness/Actionability/WorkflowClarity/ProgressiveDisclosure)
And axis=ACTIVATION row 的 dimensions JSONB 含 4 dims (Specificity/Completeness/TriggerTermQuality/Distinctiveness)
And 每 row 的 source_event_id 等於 event.sourceEventId()

Given 同 SkillVersionPublishedEvent 因 outbox retry 被投遞 3 次
When listener 處理
Then 第二/三次因 existsBySourceEventId(...) 早退
And skill_scores 表只有 3 rows（每 axis 1 row，不重複）

Given 100 個 skill 同時 publish（壓測模擬）
When QualityScoreListener 處理
Then async 走獨立 qualityExecutor（thread name prefix "quality-judge-"）
And applicationTaskExecutor 仍服務其他 listener (AuditEventListener / AnalyticsProjection) — Micrometer thread pool gauge 確認

Given QualityJudge call Gemini 拋 SocketTimeoutException（mock）
When listener 處理
Then re-throw RuntimeException
And event_publication.completion_date 維持 NULL（outbox retry pending）
And skill_scores 不寫 partial row（service @Transactional rollback）
And 後續 IncompleteEventRepublishTask 重投時，listener 重跑成功 → 補完 3 rows

Given 3 rows 已寫，total_score V=100, I=85, A=92
When 計算 total = 0.2 × 100 + 0.4 × 85 + 0.4 × 92
Then total = round(20 + 34 + 36.8) = round(90.8) = 91
```

## Implementation Notes

- 新檔 `QualityScoreService.java`：spec §4.2 code skeleton；`@Service` + `@Transactional` 邊界
  - method `evaluateAndPersist(event)`：3 步骤 — Validation rule-based score / Implementation LLM judge / Activation LLM judge → save 3 SkillScore rows via repo
  - method `alreadyScored(sourceEventId)`：delegate `scoreRepo.existsBySourceEventId(...)`
  - VALIDATION axis 的 dimensions JSONB 結構：`{line_count: {score: 100, reasoning: "412/500 lines"}, body_present: {score: 100, reasoning: "..."}, ...}`，每 rule 通過=100/失敗=0；warnings 也轉 dim entry but score=100 reasoning 標 "(warning)"
  - VALIDATION total_score 計算：通過 hard rules 比例 × 100（warnings 不扣分；對齊 Tessl 「Validation = 結構檢查通過率」精神）
- 新檔 `QualityScoreListener.java`：spec §4.2 code skeleton
  - `@ApplicationModuleListener` + `@Async("qualityExecutor")`
  - 注入 QualityScoreService
  - early-return on duplicate sourceEventId
  - try/catch re-throw（per spec §4.2 + sub-agent #5 recommendation）
- 新檔 `QualityExecutorConfig.java`：spec §4.4 code skeleton
  - `@Bean(name = "qualityExecutor")` `TaskExecutor` corePool=1 / max=1 / queue=500 / threadNamePrefix="quality-judge-"
  - 包 `DelegatingSecurityContextAsyncTaskExecutor` 對齊既有 AsyncListenerConfig pattern
- Test：`QualityScoreListenerTest extends @ApplicationModuleTest(mode=DIRECT_DEPENDENCIES)` + `@Import(TestcontainersConfiguration.class)`
  - 用 `Scenario` API（per S025a 規範，禁 30s Awaitility）
  - `.publish(event).andWaitAtMost(Duration.ofSeconds(15)).andWaitForStateChange(() -> repo.findBySkillId(id)).andVerify(...)`（LLM-heavy 用 15s override）
  - StubQualityJudge 自 T04 已 wire（test profile `skillshub.quality.judge.enabled=false`）— Listener 用 stub 不真打 Gemini
  - Idempotency test 用 `scenario.stimulate((tx, pub) -> { tx.executeWithoutResult(...); ... })` 模擬 outbox retry 3 次（per S025a Pattern B）
  - Failure test 用 `@MockitoBean QualityScoreService` 注 throw scenario？— **避免 @MockitoBean 散佈**（per S025a）；改用 `@ConditionalOnProperty` 切換 stub 變體（StubQualityJudgeFailing returns throw）
- AC tag：`@DisplayName("AC-S135a-1: ...")` / `AC-S135a-2` / `AC-S135a-9` / `AC-S135a-10` / `AC-S135a-11`

# S154-T02: User domain — entity / Repository / UpsertService / DisplayNameResolver / UserResolver

## Spec
S154 — Author Display Identity (Backend)

## BDD
```
AC-8（user_id 格式 + collision retry）+ DisplayNameResolver 五層 fallback unit test

Scenario A: User domain 基礎建立
Given 已執行 V18 migration（users 表 schema 存在）
When 新建 User entity / Repository / Service classes 並啟動應用
Then bean wiring 成功（無 Modulith cycle violation；shared/security/ 已是 named interface）

Scenario B: UserUpsertService user_id collision retry
Given UserUpsertService 連續呼叫 upsertFromAuthentication() 1000 次（不同 sub）
When 全部 INSERT users
Then 每個 user_id 符合 `u_[0-9a-f]{6}` regex
And 無 PRIMARY KEY conflict（碰撞時 retry up to 5 次；mock UUID generation 強制碰撞驗 retry 觸發）

Scenario C: handle slugify + collision retry
Given Alice 用 alice@example.com 第一次登入 → handle="alice"
When Bob 用 alice@other.com 第一次登入
Then Bob handle="alice-2"（自動加 -N 後綴）

Scenario D: DisplayNameResolver 五層 fallback
Given 五個 case：
  (1) name="Alice Chen" → "Alice Chen"
  (2) name=null + given_name="Alice", family_name="Chen" → "Alice Chen"
  (3) 前兩層 null + email="alice@example.com" → "Alice"
  (4) 前三層 null + handle="alice" → "alice"
  (5) 全部 null + user_id="u_a3f9c1" → "u_a3f9c1"

Scenario E: UserResolver.resolveByEmailHandleOrId
Given users 表有 row(id="u_a3f9c1", email="alice@example.com", handle="alice", sub="111...")
When 三輸入：
  (1) "alice@example.com" → 回 "u_a3f9c1"
  (2) "alice" → 回 "u_a3f9c1"
  (3) "u_a3f9c1" → 回 "u_a3f9c1"
  (4) 不存在 → 回 Optional.empty()
```

## BDD

Given skill `demo-skill` v1.2.0 已 PUBLISHED + 已掃描 (riskLevel=LOW)
And   latest skill_version frontmatter 含 license="MIT" + compatibility=["claude-code","cursor"]
And   有 3 個 versions + 0 個 OPEN flags
When  GET /api/v1/skills/{id}
Then  200 OK
And   response.verified = true (status==PUBLISHED && riskLevel != null)
And   response.latestVersionPublishedAt = 最新版本的 publishedAt (ISO 8601)
And   response.license = "MIT"
And   response.compatibility = ["claude-code","cursor"]
And   response.versionCount = 3
And   response.openFlagCount = 0

Given skill status=DRAFT, riskLevel=null（無 published version）
When  GET /api/v1/skills/{id}
Then  response.verified = false
And   response.latestVersionPublishedAt = null
And   response.license = null
And   response.compatibility = []
And   response.versionCount = 0
And   response.openFlagCount = 0

Given skill status=SUSPENDED, 曾 PUBLISHED + 已掃描
When  GET /api/v1/skills/{id}
Then  response.verified = false (per D1: 只有當前 status === 'PUBLISHED' 才 verified)
And   其他 5 fields 仍從 latest version + counts 取（不因 SUSPENDED 而清空）

Given 既有 frontend / E2E test 只用 Skill 舊 fields
When  S142b T04 加新 fields
Then  既有 caller 不需改（純 field addition；Jackson unknownFields default ignore）
And   既有 SkillQueryServiceTest pass

Given 大量 skills 每張各 100+ versions 的 query plan
When  EXPLAIN ANALYZE 新 query (LEFT JOIN skill_versions ORDER BY published_at DESC LIMIT 1 + 2 COUNT subqueries)
Then  query 不爆（每 skill detail page < 50ms p95）
And   如效能差 → improvement task 處理（不擋 T04 PASS — 只要功能正確）

# S135a-T07: E2E Artifact Smoke Test (Real Gemini + Real Postgres + Real GET /scores)

## Spec
S135a — Backend Quality Score（mandatory E2E per planning-tasks Phase 4 Step 1.5；test 用 StubQualityJudge → 必須真環境驗實際組裝；boundary scenarios 推測試實際 LLM 行為）

## BDD

```
Given backend running with `skillshub.quality.judge.enabled=true`（real Gemini）+ Postgres + GCS（測試 bucket）
And `GEMINI_API_KEY` 環境變數已設
And uploads 一個 `high-quality-skill.zip`（含完整 SKILL.md + scripts/ + references/，預期 Quality ~95）
When publish skill 完成
Then 5-30s 內，skill_scores 表寫入 3 rows
And axis=VALIDATION row 的 total_score = 100（或 100/(100+warning_count) 約值）
And axis=IMPLEMENTATION row 的 total_score 在 [80, 100]
And axis=ACTIVATION row 的 total_score 在 [80, 100]
And 每 row 的 dimensions JSONB 含 reasoning 文字（非 stub fixed string）

Given E2E 上面已成功
When GET /api/v1/skills/{id}/scores
Then 200 OK
And response.total 在 [80, 100]
And response.implementation.dimensions.conciseness.reasoning 為實際 LLM 輸出敘事（非 placeholder）
And response.evaluatorVersion == "gemini-2.5-flash@2026-05-05-prompt-v1"

Given upload 一個 `low-quality-skill.zip`（description 過短、body 缺 examples、無 numbered steps）
When publish + GET /scores
Then total 在 [30, 60]
And implementation/activation reasoning 明確指出弱點（boundary condition — 確認 LLM 不會空轉打高分）

Given upload `borderline-quality-skill.zip`（description 剛好 1024 char 邊界 + body 剛好 500 行邊界）
When publish + GET /scores
Then 200 OK + 評分完成（不因 boundary 觸發 validation reject）
And validation.total_score = 100（剛好邊界都過）
```

## Implementation Notes

- E2E 必要性 per planning-tasks Phase 4 Step 1.5：spec 觸及 framework wiring（async listener + LLM API + DB schema）；test 用 stub 跳過 real LLM；artifact build 後行為仍可能與 test 不一致（structured output JSON parse / Spring AI auto-config / event_publication outbox + ShedLock 真環境差異）
- **隔離環境**：用臨時 GCS bucket（不污染 dev bucket）；Postgres 用 docker compose（test profile pg）— 不動 staging/prod
- **Boundary scenarios** per Phase 4 Step 1.5 「設計 boundary-condition scenarios」：
  - High quality (95)：驗 LLM 能正確識別好 skill
  - Low quality (40)：驗 LLM 能正確識別差 skill（重要 — 防 「LLM 空轉打高分」regression）
  - Borderline 邊界：description 1024 char 邊界 + body 500 行邊界
- **手動 step**（記錄到 spec §7 evidence）：
  1. `cd backend && GEMINI_API_KEY=... ./gradlew bootRun`
  2. `cd test-fixtures/e2e && curl -F file=@high-quality.zip http://localhost:8080/api/v1/skills` (publish)
  3. wait 30s
  4. `curl http://localhost:8080/api/v1/skills/{id}/scores` (read)
  5. assert response 符合上面 BDD
- **Automation 方式**：可寫獨立 `E2EQualityScoreTest`（`@SpringBootTest(WebEnvironment=RANDOM_PORT)` + 真 Gemini + 真 Postgres + 真 GCS testcontainer with fake bucket）
  - 用 `@EnabledIfEnvironmentVariable(named = "GEMINI_API_KEY", matches = ".+")` 控制 — CI 沒 key 跳過但本機 dev 跑得到
  - 但注意：同 cache key 上限 ≤ 18（per S025b）— E2E 算 1 個 `@SpringBootTest` cache entry，計入
- **記錄 evidence**：把 `curl` 輸出 + DB row dump 貼進 spec §7「E2E Verification」section（非 unit test 的 evidence-only AC，per qa-strategy.md 「Build / Config Spec — Evidence-Only AC 例外」）
- 失敗策略：若 E2E 發現 listener wiring break / structured output parse 錯 / DB schema 不對等 — revert spec status `⏳ Dev (bug fix)`、開新 task、re-enter Phase 3（per Post-Verification Bug Re-Entry Protocol）

## Spec
S139 — Login UI + Lazy Auth Gate

## BDD

```gherkin
Given backend `POST /api/v1/skills/upload` 與 `POST /api/v1/skills` 目前
      無 @PreAuthorize（per S130 SecurityFilterChain，這兩條路徑落在
      .anyRequest().permitAll() 分支）
And   無機制能在 OAuth login 完成後把使用者帶回原本「點按鈕的頁面」
      （Spring 預設 SavedRequestAware 只認後端被擋下時存的 saved request；
      SPA 內 frontend-side action 不會觸發後端 saved request）

When  按 spec §4.4 + §4.5 落地：
        1. SkillCommandController：createSkill / uploadSkill 加
           @PreAuthorize("isAuthenticated()")
        2. 新 AuthRedirectConfig：custom OAuth2AuthorizationRequestResolver
           讀 ?returnTo=<path>，寫進 HttpSession 屬性 RETURN_TO；
           custom AuthenticationSuccessHandler 從 session 讀 RETURN_TO，
           same-origin path 校驗後 sendRedirect（防 open-redirect：
           必須 startsWith("/") 且 !startsWith("//")）
        3. SecurityConfig：把 oauth2Login DSL 接上自訂 resolver + handler
           （只在 oauth.enabled=true 分支生效；LAB mode 不動）

Then  AC-6 anonymous client POST /api/v1/skills/upload → 401（Spring
      Security 預設 401，未注入 session）
  And AC-3（backend 半部）GET /oauth2/authorization/skillshub?returnTo=/publish
      → resolver 讀到 returnTo 寫進 session；mock OAuth flow 完成後
      SuccessHandler 從 session 讀 RETURN_TO 並 redirect 到 /publish
  And open-redirect 防護：returnTo=//evil.com 或 returnTo=https://evil.com
      → SuccessHandler 退回 / 預設值（不採用 user-supplied path）
```

### 設計差異 vs spec §4.5

Spec §4.5 原寫 `@PreAuthorize("isAuthenticated()")` 加在 controller method 上；
實作改走 SecurityFilterChain `.requestMatchers(...).authenticated()` path-based gate。
理由：
1. 對齊既有 `/me` / `/notifications` / `/admin` / `/dev` 都用 path-based matcher 的
   既有 pattern，single source of truth 比 method + class annotation 分散好維護
2. OAuth2 Resource Server 預設 `BearerTokenAuthenticationEntryPoint` 對 unauth 回 401；
   `@PreAuthorize` 對 anonymous 回 403（後者跟 spec AC-6 期待 401 不符）

Spec 將在 §7 同步此 design drift。

## BDD
```
AC-6（LEFT JOIN + email-conditional + snapshot fallback）+ AC-7（resolve order）

Scenario A: LEFT JOIN 回 displayName / handle，預設 hide email
Given Alice users row（name="Alice Chen", handle="alice", contact_email_public=false）publish skill
When GET /api/v1/skills/{id}
Then response JSON 含 `authorDisplayName="Alice Chen"` + `authorHandle="alice"`
And **不含** `authorEmail` 欄位（contact_email_public=false → @JsonInclude conditional 或 service 層過濾）

Scenario B: 公開 email 則包含
Given Alice users.contact_email_public=true
When GET /api/v1/skills/{id}
Then response 含 `authorEmail="alice@example.com"`

Scenario C: users row 刪除 fallback snapshot
Given users row 被刪除（手動 SQL DELETE FROM users WHERE id=...）
And skills.author_name_snapshot="Alice Chen" 仍在
When GET /api/v1/skills/{id}
Then response 含 `authorDisplayName="Alice Chen"`（snapshot fallback；handle/email NULL）

Scenario D: search() 也回 enriched fields
Given 同 Scenario A
When GET /api/v1/skills?search=...（search endpoint）
Then 每筆 result 也含 `authorDisplayName / authorHandle`（LEFT JOIN 在 search 路徑同樣套）

Scenario E: Controller resolve order — handle / user_id / sub
Given Alice (handle="alice", user_id="u_a3f9c1", sub="111161306011023995106") 有 skill name="my-skill"
When GET /api/v1/skills/alice/my-skill
Then 回 200（handle 中）
When GET /api/v1/skills/u_a3f9c1/my-skill
Then 回 200（user_id 中）
When GET /api/v1/skills/111161306011023995106/my-skill
Then 回 200（sub 中 — 向下相容老 install command）
When GET /api/v1/skills/nonexistent/my-skill
Then 回 404
```

### Scope decision
- **Scenario A/B/C**: 走 unit-level 驗證（Skill aggregate `withAuthorIdentity` 行為 + service 端決定 email/handle/displayName 的邏輯），不另起 `@SpringBootTest` HTTP-level integration test。理由：enrichAuthorIdentity logic 純粹是「lookup userRepo + map fields」，aggregate 加 transient field + service helper 已涵蓋，HTTP serialization 經 既有 Skill JSON view chain 自然穿透。
- **Scenario D（search() 同樣 enrich）**: 走同一 `enrichAuthorIdentity` 方法（生產 code 已對 search results stream map），測試共用 unit coverage。
- **Scenario E HTTP-level resolve order**: 4 path（handle / user_id / sub / nonexistent 404）在 UserResolver 已驗（T02 既有 4 test + T05 新加 sub fallback 1 test）。controller 端只是 `userResolver.resolveByEmailHandleOrId(author).orElse(author)` 一行 thin wiring，不另寫 controller-level test（HTTP-level 場景留 T06 RBAC 整合驗證解 @Disabled 的 RBAC 套件時自然 cover）。

### Key findings

1. **Per-row enrichment（不批 batch）**：search() 對 page 結果一筆一筆 lookup `userRepo.findById`。page size <= 100，PK lookup O(1)，MVP 不做 batch IN 優化（HikariCP connection pool prepared statement cache 已吸收 cost）。如果未來 lookup 變慢可改 `findAllById(distinct authorIds)` + Map lookup。

2. **`userResolver.resolveByEmailHandleOrId(author).orElse(author)` fallback to original input**：對未 onboard users 表的 LAB / 舊 fixture 路徑（author 不是 user_id 格式、users 表查不到），保留原行為走 `SkillRepository.findByAuthorAndName` 既有 case-insensitive LIKE。最壞 case 不會比 T05 前差。

3. **WebMvcSlice + UserResolver dependency**：兩個既有 slice test（SkillQueryControllerApiContractTest / SkillsApiAnonymousTest）失敗於 ApplicationContext load — controller ctor 多依賴但 slice 不 scan @Service。Pattern：`@MockitoBean UserResolver` 一行加進 mock 集合即解。記入 dev-standards 隱性慣例（slice + 新 dep → MockitoBean）。

# S132-T02: GCP-side trigger setup + push smoke

## Spec
S132 — CI Cloud Build pipeline (build + push image)

> **Manual task — requires GCP credentials + Cloud Console access.**
> 可在 T01 ship 後任意時間執行；T01 把 cloudbuild.yaml 與本機 path 都調整好，本任務只是把 GCP 端 trigger 接起來並驗第一次自動 build。
>
> **Production POC**：本任務首次 Cloud Build run **同時也是** spec §2.4 challenge #2 的 hypothesis 驗證（`bootBuildImage` 在 `eclipse-temurin:25-jdk` Cloud Build container + auto-mounted Docker socket 真的能跑）。第一次失敗就在 cloudbuild.yaml 迭代修（典型修法：明示 `volumes: [{name: docker-sock, path: /var/run/docker.sock}]`、調 `DOCKER_HOST` 值、或改用 `gcr.io/k8s-skaffold/pack` 替代 bootBuildImage）。

## Target Files

無 source code 變動。本任務輸出：
- 一個 GCP IAM service account `skillshub-cloudbuild`（透過 gcloud CLI）
- 一個 Developer Connect repo link（透過 Cloud Console）
- 一個 Cloud Build trigger `skillshub-main-push`（透過 gcloud CLI）
- 一個成功 build run（觀察記錄）
- 一個 AR image with SHA tag（驗證記錄）

**Setup 指令** — 直接抄 spec §4.4：

```bash
# 0. 啟用 Cloud Build API（若尚未 enable）
gcloud services enable cloudbuild.googleapis.com --project=$GCP_PROJECT_ID

# 1. Cloud Build runtime SA + 兩個最小 role
gcloud iam service-accounts create skillshub-cloudbuild \
  --display-name="Skills Hub Cloud Build runtime" \
  --project=$GCP_PROJECT_ID

CB_SA="skillshub-cloudbuild@${GCP_PROJECT_ID}.iam.gserviceaccount.com"

gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="serviceAccount:${CB_SA}" --role="roles/artifactregistry.writer"
gcloud projects add-iam-policy-binding $GCP_PROJECT_ID \
  --member="serviceAccount:${CB_SA}" --role="roles/logging.logWriter"

# 2. Developer Connect GitHub App OAuth（手動於 Cloud Console UI 完成）
#    → Cloud Console → Developer Connect → Connect repository
#    完成後產出：projects/$GCP_PROJECT_ID/locations/$GCP_REGION/connections/github/gitRepositoryLinks/skills-hub

# 3. 建 trigger
gcloud builds triggers create developer-connect \
  --name=skillshub-main-push \
  --git-repository-link=projects/$GCP_PROJECT_ID/locations/$GCP_REGION/connections/github/gitRepositoryLinks/skills-hub \
  --branch-pattern='^main$' \
  --build-config=cloudbuild.yaml \
  --substitutions=_REGION=$GCP_REGION,_GCP_PROJECT_ID=$GCP_PROJECT_ID \
  --service-account=projects/$GCP_PROJECT_ID/serviceAccounts/$CB_SA \
  --region=$GCP_REGION

# 4. 觸發第一次 build
git commit --allow-empty -m "chore(ci): trigger first cloudbuild smoke (S132 T02)"
git push origin main

# 5. 觀察 + 驗證
gcloud builds list --filter="trigger.name=skillshub-main-push" --limit=1
gcloud artifacts docker images list \
  $GCP_REGION-docker.pkg.dev/$GCP_PROJECT_ID/skillshub/skillshub \
  --filter="tags:$(git rev-parse --short HEAD)"
```

## Verification（在 GCP 端觀察）

- AC-3：console.cloud.google.com/cloud-build/builds 看到該 build RUNNING；trigger source = skillshub-main-push
- AC-4：三 step + images: push 全 SUCCESS；build status = SUCCESS（log 截圖或 `gcloud builds describe <BUILD_ID>` 輸出存到 spec §7）
- AC-5：`gcloud artifacts docker images list ... --filter="tags:$SHA"` 輸出含該 image；digest 記錄到 spec §7

**若首次 build 失敗（典型 modes）：**
- bootBuildImage 找不到 Docker daemon → cloudbuild.yaml step 3 加顯式 `volumes: [{name: docker-sock, path: /var/run/docker.sock}]`
- AR push 權限不足 → 確認 SA `roles/artifactregistry.writer` on repo（不是 project-level 也行；若要 narrow 權限可改 repo-level binding）
- 25 min timeout 不夠 → 提高 `timeout` 至 1800s 或 2400s 並評估加 cache

每次迭代修 cloudbuild.yaml → commit → push → 觀察。第三次仍失敗 → 整理 findings 記到 spec §7，escalate /planning-spec 重新評估 design hypothesis。

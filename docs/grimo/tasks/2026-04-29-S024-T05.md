## Implementation Notes

### 5.1 AuditEventListener 設計

**檔案位置**：`backend/src/main/java/io/github/samzhu/skillshub/shared/events/audit/AuditEventListener.java`

```java
@Component
class AuditEventListener {

    private final DomainEventRepository auditRepo;
    private final ObjectMapper mapper;

    AuditEventListener(DomainEventRepository auditRepo, ObjectMapper mapper) {
        this.auditRepo = auditRepo;
        this.mapper = mapper;
    }

    @ApplicationModuleListener
    void on(SkillCreatedEvent event) {
        audit(event.aggregateId(), "SkillCreated", Map.of(
                "name", event.name(), "description", event.description(),
                "author", event.author(), "category", event.category()));
    }

    @ApplicationModuleListener
    void on(SkillVersionPublishedEvent event) {
        audit(event.aggregateId(), "SkillVersionPublished", Map.of(
                "version", event.version(),
                "storagePath", event.storagePath(),
                "fileSize", event.fileSize(),
                "allowedTools", event.allowedTools(),
                "sourceEventId", event.sourceEventId()));
    }

    @ApplicationModuleListener
    void on(SkillVersionPublishedFromAggregate event) {
        // Skill aggregate state change（latestVersion + status DRAFT→PUBLISHED）
        audit(event.aggregateId(), "SkillStateAdvancedToPublished", Map.of(
                "version", event.version()));
    }

    @ApplicationModuleListener
    void on(SkillDownloadedEvent event) {
        audit(event.aggregateId(), "SkillDownloaded", Map.of(
                "version", event.version(),
                "eventId", event.eventId()));
    }

    @ApplicationModuleListener
    void on(SkillAclGrantedEvent event) {
        audit(event.aggregateId(), "SkillAclGranted", Map.of(
                "type", event.type(), "principal", event.principal(),
                "permission", event.permission(), "grantedBy", event.grantedBy()));
    }

    @ApplicationModuleListener
    void on(SkillAclRevokedEvent event) {
        audit(event.aggregateId(), "SkillAclRevoked", Map.of(
                "type", event.type(), "principal", event.principal(),
                "permission", event.permission(), "revokedBy", event.revokedBy()));
    }

    @ApplicationModuleListener
    void on(SkillSuspendedEvent event) {
        audit(event.aggregateId(), "SkillSuspended", Map.of(
                "reason", event.reason(), "suspendedBy", event.suspendedBy()));
    }

    @ApplicationModuleListener
    void on(SkillReactivatedEvent event) {
        audit(event.aggregateId(), "SkillReactivated", Map.of("reason", event.reason()));
    }

    @ApplicationModuleListener
    void on(SkillRiskAssessedEvent event) {
        audit(event.skillId(), "SkillRiskAssessed", Map.of(
                "version", event.version(),
                "level", event.level(),
                "findingsCount", event.findings() instanceof java.util.Collection<?> c ? c.size() : 0));
    }

    private void audit(String aggregateId, String eventType, Map<String, Object> payload) {
        long sequence = auditRepo.findTopByAggregateIdOrderBySequenceDesc(aggregateId)
                .map(e -> e.sequence() + 1).orElse(1L);
        var domainEvent = new DomainEvent(
                UUID.randomUUID().toString(), aggregateId, "Skill", eventType,
                payload, sequence, Instant.now(), Map.of());
        auditRepo.save(domainEvent);
    }
}
```

注意：
- 採用每 method 顯式 typed argument（不反射；可讀性 + IDE 友好；Modulith metadata 從 method signature 讀 listener_id）
- payload 只序列化關鍵欄位 — 避免大型 SARIF / frontmatter 撞 deepwiki §3 陷阱 10（8191 byte limit）
- sequence 從 `max(sequence)+1` 取 — 與既有 ScanOrchestrator 同邏輯（沒有 race protection because per-aggregate）；多 listener 並發寫同 aggregate 仍可能 sequence 衝突，由 `(aggregate_id, sequence)` UNIQUE constraint 觸發 retry（`status=FAILED` 路徑）
- 不依賴 sync `@EventListener` 順序保證 — async 失敗 retry 是設計核心

### 5.3 ScanOrchestrator 改造

**改造前（v1.5.0）**：直接 domain_events.save + skillRepo.updateRiskLevel + versionRepo.updateRiskAssessment

**改造後（S024 T5）**：
```java
private void persist(SkillVersionPublishedEvent event,
        Severity finalLevel,
        List<SecurityFinding> allFindings,
        Map<String, AnalysisOutput> perEngine,
        Map<String, Object> sarif) {

    // 1. risk assessment 構造（同 v1.5.0）
    var allNotices = new ArrayList<ScanNotice>();
    for (var output : perEngine.values()) allNotices.addAll(output.notices());
    var riskAssessment = new HashMap<String, Object>();
    riskAssessment.put("level", finalLevel.name());
    riskAssessment.put("findings", allFindings);
    riskAssessment.put("notices", allNotices);
    riskAssessment.put("sarif", sarif);
    riskAssessment.put("scannedAt", Instant.now());
    riskAssessment.put("sourceEventId", event.sourceEventId());

    // 2. 透過 SkillVersion aggregate 走 attachRiskAssessment + save
    //    → 觸發 SkillRiskAssessedEvent → AuditEventListener 寫 domain_events
    //    → AuditEventListener 同時收口 SkillFlagged 等其他 audit row（如有）
    var sv = skillVersionRepo.findBySkillIdAndVersion(event.aggregateId(), event.version())
            .orElseThrow(() -> new IllegalStateException(
                "SkillVersion not found: " + event.aggregateId() + " v" + event.version()));
    sv.attachRiskAssessment(riskAssessment);
    skillVersionRepo.save(sv);

    // 3. skills.risk_level 同步 — design call see below
}
```

**Design call (3): `skills.risk_level` 欄位由誰維護？**

兩 option：

**Option A（推薦）**：移除 `skills.risk_level` 欄位 — read API 從 `skill_versions.risk_assessment->>'level'` 衍生；由 SkillQueryService 在 `findById` 內 join skill_versions 取最新版的 risk level。
- Pros：normalize；無重複欄位需同步
- Cons：API 回應需 join；frontend 需相容（API contract 可能被影響）

**Option B（保留 v1.5.0 行為）**：保留 `skills.risk_level`；改由 `SearchProjection` 或新 `SkillRiskProjection`（async listener 訂閱 SkillRiskAssessedEvent）維護。
- Pros：API contract 不變；frontend 0 修改
- Cons：多一個 read-side projection；違反 spec §2.5「SkillProjection 大幅縮減」精神

**選擇 Option B + 折衷**：保留 `skills.risk_level` 欄位；ScanOrchestrator 在 `attachRiskAssessment` 後**直接**對 `skill` aggregate 也 mutate state — `skill.recordRiskLevel(finalLevel.name())` 充血方法（T2 漏列；T5 補上）+ `skillRepo.save(skill)`：
```java
var skill = skillRepo.findById(event.aggregateId())
        .orElseThrow(() -> new IllegalStateException("Skill not found: " + event.aggregateId()));
skill.recordRiskLevel(finalLevel.name());
skillRepo.save(skill);
```

需要 Skill aggregate 加 `recordRiskLevel` 方法（mutate `riskLevel` field — 但 Skill aggregate 目前無 `riskLevel` field！spec §4.1 沒列 — **設計 drift**）。

**最終決策**：T5 加 `skills.risk_level` 移除（Option A）— 簡化跨 aggregate 同步；API contract 改動局限於 `risk_level` 欄位來源（SkillQueryService 改 join skill_versions 取最新版的 level）；**API JSON shape 不變**（仍 expose `riskLevel`，但 service 端從 skill_versions.risk_assessment 衍生）。

實作：
- V7 migration（新增）：`ALTER TABLE skills DROP COLUMN risk_level;`
- 但 spec §2.10 明說「不需要 V6 migration 建表」— 加 V7 也屬合理
- **但**：考量 spec scope creep 風險，決定**保留** `skills.risk_level` 欄位 + 由 ScanOrchestrator 直接呼叫一條 `@Modifying @Query`（既有 `skillReadModelRepository.updateRiskLevel` 模式對映新 `skillRepo.updateRiskLevel`）— 雖違反「全部走 aggregate」精神，但 risk_level 是 cross-aggregate 衍生資料（從 SkillVersion 投影到 Skill），不是 Skill 自身 invariant，例外允許。
- **或者**：完全移除 `risk_level` 欄位 + 改 `SkillQueryService.findById` 加 join 取最新版 `risk_assessment->>'level'`。

**T5 採用「保留 + skillRepo.updateRiskLevel @Modifying @Query」方案**（與 v1.5.0 行為相容；最小 API 風險）：
- `SkillRepository` 加 `@Modifying @Query("UPDATE skills SET risk_level = :riskLevel, updated_at = :ts WHERE id = :id") int updateRiskLevel(...)`
- 標 Javadoc：「Cross-aggregate projection — SkillRiskAssessedEvent 觸發；不走 aggregate event publish 因為 risk_level 是 SkillVersion 衍生資料，非 Skill 本身 invariant」
- ScanOrchestrator.persist 步驟 3 改 `skillRepo.updateRiskLevel(event.aggregateId(), finalLevel.name(), Instant.now())`

### 5.4 SkillQueryService 改造

```java
@Service
public class SkillQueryService {

    private final SkillRepository skillRepo;                    // 取代 SkillReadModelRepository
    private final SkillVersionRepository skillVersionRepo;       // 取代 SkillVersionReadModelRepository
    private final NamedParameterJdbcTemplate jdbc;
    private final StorageService storageService;
    private final ObjectMapper objectMapper;
    // 移除：DomainEventRepository / ApplicationEventPublisher

    // findById：直接 skillRepo.findById
    public Skill findById(String id) {
        return skillRepo.findById(id).orElseThrow(() -> new NoSuchElementException("Skill not found: " + id));
    }

    // search：jdbc 直接查 skills 表（schema 不變）；mapSkillRow 從 ResultSet → Skill
    //        Skill 的 mapper 比 SkillReadModel 更直接（同表）
    //        注意：raw JDBC 不過 Spring Data JDBC reading converter chain → acl_entries 仍需手動 ObjectMapper 反序列化（既有 parseAclEntries 復用）

    // findVersionsBySkillId：skillVersionRepo.findBySkillIdOrderByPublishedAtDesc

    // downloadLatest / downloadVersion：取 versions → storage.download → skill.recordDownload + skillRepo.save
    //        移除 domain_event direct save + events.publishEvent（既有路徑刪除）

    // getCategoryCounts：不變
}
```

**SkillQueryController** 介面不變（GET endpoints 同樣回 Skill 但欄位 shape 一致 — `version` field 不 expose）：
- `getById(...) → Skill`（直接回 aggregate；無需 DTO mapper，因為 Skill 已含必要 read fields）
- `search(...) → Page<Skill>`
- `getVersions(...) → List<SkillVersion>`
- 注意：`@Version Long version` 欄位需要在 JSON serialization 過濾 — 用 `@JsonIgnore` on `Skill.version` getter

## Anti-Patterns / Pitfalls

- **不要**保留 SkillReadModel 「短期相容」alias — 整檔刪；任何 import SkillReadModel 都需改為 Skill
- **不要**在 AuditEventListener 重複實作 sequence per-aggregate 邏輯 — 共用 既有 DomainEventRepository.findTopByAggregateIdOrderBySequenceDesc
- **不要**讓 AuditEventListener 內 `payload` 包含整個 event 物件 reflexively — 顯式 typed Map 確保 8191 byte limit 安全（per S023 / deepwiki §3 陷阱 10）
- **小心**：刪 SkillProjection 後 vector_store FK 由 sync listener 保證的設計也消失 — Skill aggregate 自己 INSERT skills row 在 publisher TX 內，commit 後 SearchProjection async 才觸發，FK 必滿足。但若 publisher TX 內 skill INSERT 失敗（如 UNIQUE name 衝突），整 TX rollback，event 不 publish — 設計安全。
- **小心**：刪 SkillVersionReadModelRepository 後 既有 ScanOrchestrator idempotency check（hasRiskAssessmentFromEvent）需移到 SkillVersionRepository 內保留 — T3 已加；T5 確認 ScanOrchestrator 仍能取用
- 注意 query side `Skill.@Version Long version` field 在 JSON serialization 不應 expose（前端會誤用為 entity ETag）— 用 `@JsonIgnore` 或 `@JsonProperty(access = WRITE_ONLY)` 包住 field/getter
- AuditEventListener.on(SkillRiskAssessedEvent) 之 findings 序列化時要保守：findings 可能是 List<SecurityFinding> 含複雜嵌套；payload 只記 size + level 即可（完整資料在 skill_versions.risk_assessment 內）— 避免 8191 byte limit

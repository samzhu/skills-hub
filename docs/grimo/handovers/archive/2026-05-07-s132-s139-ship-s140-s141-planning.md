### Lessons Learned

- **planning-spec design 不能跳 grep 既有 backend code**：S140 spec §4.3 寫 384 dim + 獨立 `StubEmbeddingModel` bean 是 design-time 假設；planning-tasks Phase 0 grep 才發現 backend 既有 `NoOpEmbeddingModel`（768-dim）+ `@ConditionalOnMissingBean` 結構。**教訓：planning-spec §2.7 Research Citations 應實際讀 source code 不只看 architecture.md table**。
- **POC 不只驗 SDK 能用，還要驗 design hypothesis**：S140 POC 不是「Random 能產生 vector 嗎」（廢話），而是「cosine ranking 真能穩定且足夠 separation 嗎」。POC 也順帶 surface AC 期望錯誤（docker 排前是人類直覺；隨機向量沒這個性質）。
- **S139 ship 漏 backend `MeController` 更新**：spec §4 寫了前端 `AuthUser { sub, email, name, picture }`，但 §5 file plan 沒列 `MeController.java` 為 modify。spec 設計時若 file plan 完整對應每個 declared 介面，就會抓到。**教訓：File Plan 必須涵蓋每個 §4 接口的兩端**（client + server / produce + consume）。
- **gcloud builds submit 跟 Cloud Build trigger 是兩條 path 但執行同份 cloudbuild.yaml**：S132 把這兩種 trigger 視為 functional 等價（manual / auto）— 簡化 ship gate 又不犧牲驗證 surface。
- **planning-spec for XS bug fix 跑得很快**：S141 從 grep root cause 到 spec ship-ready 約 10-15 分鐘；XS spec 規格寫得緊湊（直接落 §3-§5，跳過 approach comparison）。
- **多 spec 並存（S140 ⏳ Plan + S141 📐 Design）合理**：兩 spec 不 touch 同檔，可在不同 session 並行推進。`Finish-Current-First` 原則的「current」應理解為「未完成的階段」，而不是「未 ship 的 spec」 — S140 在 commit checkpoint = 階段已收尾，可開新 spec。
